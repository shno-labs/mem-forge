Repository Identity vs. Project Mapping

repo_identifier is the stable context key for coding-session memories. It is stored on memory rows and vector metadata as a ranking/grouping signal, not an access-control signal.

The Codex and Claude Code local adapters send this normalized remote-derived identity in the upload payload's repo field. The service normalizes again before storing repo_identifier, so older plugins that still send a plain slug continue to land on a deterministic key.

🔍 repo_identifier derivation

  • Normalized VCS remote URL (e.g. github.com/org/repo)
  • Explicit repo slug from client
  • Existing receipt repo value
  • Workspace basename (display fallback only)

📂 Project mapping

Project mapping is still relevant but not the primary session identity.

  • Confluence page tree, Jira filter: natural project mapping
  • Coding sessions: repo is more stable than project
  • No project mapping: memories land in UNSORTED

Memory Levels and Curation Lineage

Two memory levels, both searchable. Consolidated memories do not replace atomic memories in V1.

● Atomic memory

Extracted from one source item or session package. The base unit of memory.

● Consolidated memory

Created by the curator from multiple atomic memories. Has explicit lineage back to its sources.

Curator Design

⚙ MemoryCuratorRunner

  • Receives candidate memories
  • Filters through registered policy
  • Groups into deterministic clusters
  • Creates consolidated memory drafts
  • Inserts consolidated memories
  • Records derivation lineage
  • Records memory_curation_runs audit data

📜 V1 policy: agent_session.coding.v1

  • Clients: codex, claude-code
  • Memory level: atomic only
  • Required: repo_identifier
  • Cluster boundary: visibility + owner + repo + project + topic

No generic fallback curator. Other source types require explicit policy and tests.

Curator safety rules

  • Do not consolidate across private owners
  • Do not consolidate private and workspace-visible memories together
  • Do not consolidate across repositories
  • Do not mutate or delete atomic memories in V1
  • Keep source provenance and derivation edges
  • Store memory_curation_runs metadata for auditability
  • Treat generated agent-session packages as low-authority source material

Search Behavior

Search remains access-controlled by existing rules. repo_identifier only affects relevance, not access.

1Retrieve normal candidates (visibility, owner, workspace, status, project rules apply)
2Enrich candidates with repo_identifier, memory_level, and curation_cluster_id
3Group candidates by curation family
4If consolidated + children both present, prefer consolidated by default
5Keep a child result when it strongly outranks the summary (exact error, file, symbol, issue-specific query)

When search receives active_repo_identifier, same-repo memories get an affinity boost. Cross-repo results are not hidden.

User-Visible Behavior

There is no new UI surface in this round. Changes are indirect only.

✓ Users may notice

  • Coding-agent search results become more repo-relevant
  • Broad searches show fewer duplicate session memories once curated summaries exist
  • MCP and hook context becomes less noisy for long-running repo work

✕ Users should not see

  • A new curator dashboard
  • New source configuration fields
  • A different source-list workflow
  • Automatic deletion or disappearance of atomic memories
